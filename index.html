Module 1

Closure

1.1 Execution Context & Call Stack

Cơ bản

Mỗi khi JavaScript chạy một function, engine tạo ra một Execution Context (EC) — một "hộp cát" chứa: biến cục bộ, tham chiếu đến scope cha, và this. Tất cả EC được quản lý bởi Call Stack theo nguyên tắc LIFO.

JavaScript const x = 10; // Global EC: x = 10 function outer() { const y = 20; // outer() EC: y = 20 function inner() { const z = 30; // inner() EC: z = 30 console.log(x + y + z); // 60 — truy cập được cả 3 scope } inner(); // push inner EC → pop sau khi xong } outer(); // push outer EC → pop sau khi xong
Thứ tự Call Stack: Global EC luôn ở đáy. Mỗi function call push một EC mới lên top. Khi function kết thúc, EC bị pop — biến bên trong bị hủy (trừ khi closure giữ lại).
Thời điểmCall StackEC hiện tại
Khởi động[Global]Global EC: x=10
outer() được gọi[Global, outer]outer EC: y=20
inner() được gọi[Global, outer, inner]inner EC: z=30
inner() xong[Global, outer]z bị hủy
outer() xong[Global]y bị hủy

1.2 Scope & Scope Chain

Cơ bản

Lexical scope — scope được xác định tại lúc viết code, không phải lúc chạy. JavaScript nhìn vào nơi function được định nghĩa để tìm biến.

Scope chain — khi tìm biến, JS tìm từ scope hiện tại → lên scope cha → lên tiếp... cho đến Global. Nếu không tìm thấy → ReferenceError.

// Lexical scope — quan trọng: nơi ĐỊNH NGHĨA, không phải nơi GỌI const name = 'Global'; function greet() { console.log(name); // 'Global' — đọc từ scope cha (lexical) } function wrapper() { const name = 'Local'; greet(); // vẫn in 'Global' — greet được định nghĩa ở Global scope! } wrapper(); // → 'Global' (không phải 'Local')
Quy tắc tìm biến trong scope chain: inner → outer → global → undefined (hoặc ReferenceError với strict mode).

1.3 Closure — Bản chất

Trung cấp

Định nghĩa chính xác: Closure là khả năng một function ghi nhớ và tiếp cận lexical scope của nó ngay cả khi function đó đang thực thi bên ngoài scope ban đầu.

Khi outer() kết thúc, EC của nó bị pop khỏi Call Stack. Nhưng nếu hàm bên trong vẫn còn tham chiếu đến biến của outer — JavaScript không hủy những biến đó. Chúng sống trong Closure Environment.

function makeCounter() { let count = 0; // biến "sống" trong closure return function() { count++; return count; }; } const counter = makeCounter(); // makeCounter() đã kết thúc — nhưng count vẫn còn sống! counter(); // → 1 counter(); // → 2 counter(); // → 3 // Mỗi lần gọi makeCounter() tạo Closure Environment RIÊNG: const c1 = makeCounter(); const c2 = makeCounter(); c1(); c1(); // c1 count = 2 c2(); // c2 count = 1 — hoàn toàn độc lập!
Closure = function + environment: Mỗi lần gọi makeCounter(), một Closure Environment mới được tạo với biến count riêng biệt. c1 và c2 không chia sẻ state với nhau.

1.4 Closure trong vòng lặp — Lỗi kinh điển

Trung cấp

Đây là bẫy phỏng vấn số 1 về closure. Vấn đề xảy ra khi dùng var trong for loop.

❌ Sai — dùng var
var funcs = []; for (var i = 0; i < 3; i++) { funcs[i] = function() { return i; // var dùng chung! }; } funcs[0](); // → 3 (không phải 0!) funcs[1](); // → 3 funcs[2](); // → 3
✅ Fix 1 — dùng let
const funcs = []; for (let i = 0; i < 3; i++) { funcs[i] = function() { return i; // let: binding riêng mỗi iteration }; } funcs[0](); // → 0 ✓ funcs[1](); // → 1 ✓ funcs[2](); // → 2 ✓
✅ Fix 2 — IIFE
var funcs = []; for (var i = 0; i < 3; i++) { funcs[i] = (function(j) { // j là parameter — copy của i tại thời điểm này return function() { return j; }; })(i); // gọi ngay, truyền i vào }
✅ Fix 3 — Factory function
function createFunc(n) { return function() { return n; }; } var funcs = []; for (var i = 0; i < 3; i++) { funcs[i] = createFunc(i); } // Cách sạch và dễ đọc nhất trong thực tế
Tại sao var bị lỗi? var không có block scope — tất cả 3 function đều closure over cùng một biến i. Khi vòng lặp kết thúc, i = 3. Khi gọi bất kỳ function nào → đọc i lúc đó = 3. let tạo một binding mới cho mỗi iteration nên mỗi function có i riêng.

1.5 Factory Function Chuyên Sâu

Nâng cao

Factory function không chỉ là "hàm trả về object" — nó là cách tạo nhiều instance độc lập, mỗi cái có closure riêng, thay thế được class trong nhiều tình huống.

// Pattern cơ bản — Private state + Public interface function createUser(name, role) { let loginCount = 0; // PRIVATE — chỉ closure thấy const createdAt = new Date(); // PRIVATE return { name, role, // PUBLIC login() { loginCount++; return `${name} login lần #${loginCount}`; }, getStats: () => ({ loginCount, createdAt }) }; } const alice = createUser('Alice', 'admin'); const bob = createUser('Bob', 'user'); alice.login(); // → "Alice login lần #1" alice.login(); // → "Alice login lần #2" bob.login(); // → "Bob login lần #1" (độc lập!) alice.loginCount; // → undefined — KHÔNG truy cập được trực tiếp!

Factory vs Class — Vấn đề this

❌ Class — this binding phức tạp
class Counter { constructor() { this.n = 0; } inc() { this.n++; } } const c = new Counter(); const { inc } = c; inc(); // 💥 TypeError: this là undefined // Phải bind: c.inc.bind(c)
✅ Factory — không cần this
function createCounter() { let n = 0; return { inc: () => ++n, // arrow fn dùng closure get: () => n }; } const { inc } = createCounter(); inc(); inc(); // ✅ Hoạt động bình thường

Composition — ghép khả năng thay vì kế thừa

// Các "ability" riêng lẻ: const canSwim = (state) => ({ swim: () => `${state.name} đang bơi` }); const canFly = (state) => ({ fly: () => `${state.name} đang bay` }); const canDive = (state) => ({ dive: () => `${state.name} đang lặn` }); // Compose tùy ý — không cần kế thừa: function createDuck(name) { const state = { name }; return Object.assign({}, canSwim(state), canFly(state), canDive(state)); } function createPenguin(name) { const state = { name }; return Object.assign({}, canSwim(state), canDive(state)); // không có fly! }

Partial Application

function partial(fn, ...presetArgs) { return function(...laterArgs) { // closure giữ presetArgs return fn(...presetArgs, ...laterArgs); }; } const multiply = (a, b) => a * b; const double = partial(multiply, 2); const triple = partial(multiply, 3); double(5); // → 10 triple(5); // → 15 // Real-world: API caller cấu hình sẵn const getUser = partial(apiCall, 'GET', '/users'); const postUser = partial(apiCall, 'POST', '/users');

1.6 Memoization & Revealing Module Pattern

Nâng cao
// Memoization — cache kết quả tính toán function memoize(fn) { const cache = new Map(); // closure giữ cache sống mãi return function(...args) { const key = JSON.stringify(args); if (cache.has(key)) { return cache.get(key); // cache hit — trả ngay! } const result = fn(...args); cache.set(key, result); return result; }; } const expensiveFib = memoize(function fib(n) { return n <= 1 ? n : expensiveFib(n-1) + expensiveFib(n-2); }); console.log(expensiveFib(40)); // lần đầu: tính console.log(expensiveFib(40)); // lần sau: cache hit, tức thì!
// Revealing Module Pattern — tạo module với public/private API const BankAccount = (() => { let balance = 0; // PRIVATE const history = []; // PRIVATE function validate(amount) { if (amount <= 0) throw new Error('Số tiền không hợp lệ'); } return { // PUBLIC interface deposit(amount) { validate(amount); balance += amount; history.push({ type: '+', amount, balance }); }, withdraw(amount) { validate(amount); if (amount > balance) throw new Error('Không đủ tiền'); balance -= amount; history.push({ type: '-', amount, balance }); }, getBalance: () => balance, getHistory: () => [...history] // trả bản sao — không expose trực tiếp }; })(); // IIFE — gọi ngay BankAccount.deposit(1000000); BankAccount.withdraw(300000); BankAccount.getBalance(); // → 700000 BankAccount.balance; // → undefined — không thể truy cập!

Promise & Async

2.1 Event Loop — Trái tim của JavaScript Async

Cơ bản

JavaScript là single-threaded — chỉ làm một việc tại một thời điểm. Event Loop là cơ chế phối hợp giữa Call Stack, Microtask Queue, và Macrotask Queue để xử lý async code.

console.log('1 - sync'); setTimeout(() => console.log('2 - macro'), 0); Promise.resolve().then(() => console.log('3 - micro')); console.log('4 - sync'); // Output: 1 → 4 → 3 → 2 // Tại sao? Thứ tự ưu tiên: Sync → Microtask → Macrotask
BướcHành độngCall StackMicro QueueMacro QueueOutput
1log('1')[log][][]1
2setTimeout[setTimeout][][cb_2]
3Promise.then[then][cb_3][cb_2]
4log('4')[log][cb_3][cb_2]4
5Stack trống → chạy Micro[cb_3][][cb_2]3
6Micro trống → chạy Macro[cb_2][][]2
Quy tắc vàng: Microtask Queue (Promise .then, queueMicrotask, MutationObserver) luôn được drain hoàn toàn trước khi Event Loop lấy task tiếp theo từ Macrotask Queue (setTimeout, setInterval, I/O events).

2.2 Callback Hell — Vấn đề cần giải quyết

Cơ bản
❌ Callback Hell — "Pyramid of Doom"
getUser(id, function(err, user) { if (err) throw err; getPosts(user.id, function(err, posts) { if (err) throw err; getComments(posts[0].id, function(err, comments) { if (err) throw err; saveLog(comments, function(err) { if (err) throw err; // ... tiếp tục sâu hơn }); }); }); });
✅ async/await — Đọc như code đồng bộ
async function loadData(id) { try { const user = await getUser(id); const posts = await getPosts(user.id); const comments = await getComments(posts[0].id); await saveLog(comments); } catch (err) { // Một chỗ xử lý tất cả lỗi console.error(err); } }

2.3 Promise — 3 Trạng thái

Trung cấp

Một Promise là object đại diện cho một giá trị chưa có ngay. Nó có đúng 3 trạng thái và chỉ chuyển trạng thái một lần duy nhất, không thể quay lại.

⏳ Pending
Đang chờ — không thể đọc value
→
✅ Fulfilled
resolve(value) — .then() chạy
hoặc
❌ Rejected
reject(error) — .catch() chạy
// Tạo Promise thủ công const p = new Promise((resolve, reject) => { setTimeout(() => { const ok = Math.random() > 0.5; if (ok) resolve('Thành công!'); else reject(new Error('Thất bại!')); }, 1000); }); // .then() / .catch() / .finally() p .then(value => { console.log('OK:', value); return doNextThing(value); // có thể chain tiếp }) .catch(err => { console.error('Lỗi:', err.message); }) .finally(() => { console.log('Luôn chạy — dù thành công hay thất bại'); }); // ⚠️ Quan trọng: fetch() không tự throw khi HTTP 4xx/5xx! const res = await fetch('/api/users'); if (!res.ok) throw new Error(`HTTP Error: ${res.status}`); // phải tự check! const data = await res.json();

2.5 async / await — Chuyên Sâu

Nâng cao

async/await là syntactic sugar — bên dưới vẫn là Promise. Một async function luôn trả về Promise. await không block JS engine — nó nhường control về Event Loop và đăng ký phần còn lại vào Microtask Queue.

// Cơ chế: await nhường control async function demo() { console.log('A'); // sync await Promise.resolve(); // nhường control về Event Loop console.log('C'); // chạy sau B! } demo(); console.log('B'); // chạy trước C // Output: A → B → C

Error Handling đúng chuẩn

❌ 3 lỗi phổ biến
// Lỗi 1: không catch async function bad1() { const data = await fetch('/api'); } // 💥 UnhandledPromiseRejection // Lỗi 2: nuốt lỗi async function bad2() { try { await fetch('/api'); } catch (e) { console.log(e); } // caller không biết có lỗi! } // Lỗi 3: return (thiếu await) async function bad3() { try { return fetch('/api'); // thiếu await! } catch (e) { // không catch được lỗi từ fetch! } }
✅ Đúng chuẩn
async function fetchUser(id) { try { const res = await fetch(`/users/${id}`); // fetch() chỉ reject khi network lỗi // HTTP 404/500 vẫn resolve! Phải tự check: if (!res.ok) throw new Error(`HTTP ${res.status}`); return await res.json(); // return AWAIT! } catch (err) { console.error('fetchUser:', err); throw err; // rethrow cho caller! } }

Sequential vs Parallel — Sai lầm tốn kém nhất

❌ Sequential — lãng phí 4.5s
async function loadSlow() { const user = await getUser(); // 1s const posts = await getPosts(); // 2s const tags = await getTags(); // 1.5s // Tổng: 4.5s — dù không liên quan nhau! }
✅ Parallel — chỉ 2s
async function loadFast() { const [user, posts, tags] = await Promise.all([ getUser(), getPosts(), getTags() ]); // Tổng: max(1, 2, 1.5) = 2s }

Pitfalls — await trong forEach

const ids = [1, 2, 3]; // ❌ forEach KHÔNG await — tất cả chạy song song, thứ tự không đảm bảo: ids.forEach(async (id) => { const u = await getUser(id); console.log(u); // thứ tự ngẫu nhiên! }); // ✅ for...of — await tuần tự đúng thứ tự: for (const id of ids) { const u = await getUser(id); console.log(u); // 1 → 2 → 3 đúng thứ tự } // ✅ map + Promise.all — song song nhưng giữ thứ tự: const users = await Promise.all(ids.map(id => getUser(id))); console.log(users); // [User1, User2, User3] đúng thứ tự, chạy song song

Thực chiến

3.1 Closure trong Async — Race Condition & Stale Closure

Nâng cao

Đây là bug tinh vi nhất — xảy ra khi closure "giữ" một biến nhưng biến đó thay đổi trước khi async hoàn thành.

Stale Closure Bug

// ❌ Bug: closure capture giá trị tại thời điểm tạo function setupAutoSave() { let count = 0; setInterval(async () => { // await saveToServer() mất 500ms await saveToServer(count); // count có thể đã thay đổi khi await xong! }, 1000); document.onclick = () => count++; } // ✅ Fix: dùng ref object — closure capture REFERENCE, không phải value function setupAutoSaveFixed() { const state = { count: 0 }; // Object reference không bị stale! setInterval(async () => { await saveToServer(state.count); // luôn đọc giá trị mới nhất }, 1000); document.onclick = () => state.count++; }

Race Condition — Request cũ về sau request mới

❌ Race Condition
async function search(query) { const results = await fetchSearch(query); // "cat" chậm hơn "cats" // → "cats" hiện kết quả đúng // → "cat" về sau, ghi đè = sai! showResults(results); // 💥 }
✅ Fix 1 — AbortController
let controller = null; async function searchFixed(query) { controller?.abort(); // huỷ request trước controller = new AbortController(); try { const res = await fetch(url, { signal: controller.signal }); showResults(await res.json()); } catch (e) { if (e.name !== 'AbortError') throw e; } }
// ✅ Fix 2 — Request ID pattern (dùng closure) let latestId = 0; // closure variable async function searchFixed2(query) { const thisId = ++latestId; // capture ID tại thời điểm gọi const results = await fetchSearch(query); // Chỉ hiển thị nếu đây vẫn là request mới nhất: if (thisId === latestId) { showResults(results); // ✅ an toàn } // Request cũ hơn → latestId đã tăng → âm thầm bỏ qua }
Tại sao Fix 2 hoạt động? thisId được capture (closure) tại thời điểm mỗi lần gọi search(). Sau khi await xong, so sánh với latestId — nếu đã có request mới hơn, latestId đã tăng lên và request cũ biết mình "lỗi thời" rồi tự bỏ qua.

3.2 Retry Pattern với Exponential Backoff

Nâng cao
function withRetry(fn, maxRetries = 3, baseDelay = 100) { return async function(...args) { // closure giữ fn, maxRetries, baseDelay let attempt = 0; while (attempt < maxRetries) { try { return await fn(...args); // thành công → trả kết quả } catch (err) { attempt++; if (attempt === maxRetries) throw err; // hết lần thử → throw // Exponential backoff: 100ms → 200ms → 400ms → ... const delay = Math.pow(2, attempt) * baseDelay; await new Promise(r => setTimeout(r, delay)); } } }; } // Sử dụng: const fetchWithRetry = withRetry(fetch, 3); const res = await fetchWithRetry('/api/users');

3.3 Debounce & Throttle từ đầu

Nâng cao
Quy tắc phân biệt: Debounce = "chờ yên X ms rồi mới chạy" (mỗi lần gọi reset timer). Throttle = "tối đa chạy 1 lần mỗi X ms" (bỏ qua nếu chưa đủ thời gian).
⏳ Debounce — chờ yên rồi mới chạy
function debounce(fn, delay) { let timer; // closure! return function(...args) { clearTimeout(timer); timer = setTimeout(() => { fn.apply(this, args); }, delay); }; } // Dùng cho: search input, form validation const onSearch = debounce( (q) => callAPI(q), 500 ); // Gõ liên tục → chỉ gọi 1 lần sau 500ms yên
⏱ Throttle — tối đa 1 lần/Xms
function throttle(fn, limit) { let lastRun = 0; // closure! return function(...args) { const now = Date.now(); if (now - lastRun >= limit) { lastRun = now; fn.apply(this, args); } // chưa đủ thời gian → bỏ qua }; } // Dùng cho: scroll, resize, mouse move const onScroll = throttle( () => updateUI(), 300 ); // Scroll liên tục → chạy tối đa 3.3 lần/giây
Tình huốngDùng gìLý do
Search input, autocompleteDebounceChờ user gõ xong mới gọi API
Form validationDebounceValidate sau khi user dừng nhập
Window scroll handlerThrottleUpdate UI đều đặn, không bỏ sót
Window resizeThrottle hoặc DebounceThrottle nếu cần cập nhật liên tục; Debounce nếu chỉ cần kết quả cuối
Mouse move / dragThrottleCần update đều, không bỏ sót frame
Nút Submit / SaveDebounceTránh double-submit
Realtime analytics trackingThrottleGửi event đều đặn, không spam

3.4 LRU Cache với TTL — Implementation Đúng

Nâng cao
Lỗi phổ biến khi implement LRU: Dùng Map nhưng không promote key sau mỗi lần get() hoặc set() → thực chất là FIFO, không phải LRU. JavaScript Map giữ thứ tự insertion, vì vậy: delete rồi re-insert = move về cuối (MRU).
function createCache({ ttl = 10000, maxSize = 100 } = {}) { const store = new Map(); // CLOSURE — private, không ai truy cập trực tiếp let hits = 0, misses = 0; const isExpired = e => Date.now() - e.createdAt > ttl; // ✅ Promote = delete + re-insert = key về cuối Map (MRU) function promote(key) { const entry = store.get(key); store.delete(key); store.set(key, entry); // về cuối = MRU } return { get(key) { const e = store.get(key); if (!e || isExpired(e)) { store.delete(key); misses++; return null; // miss hoặc expired } promote(key); // ✅ Cốt lõi LRU: move lên MRU sau mỗi get() hits++; return e.value; }, set(key, value) { if (store.has(key)) { store.delete(key); // ✅ Xoá vị trí cũ để re-insert về MRU } else if (store.size >= maxSize) { // ✅ Evict đúng LRU = phần tử đầu tiên của Map const lruKey = store.keys().next().value; store.delete(lruKey); } store.set(key, { value, createdAt: Date.now() }); }, invalidate(key) { store.delete(key); }, clear() { store.clear(); hits = 0; misses = 0; // ✅ Reset stats — lỗi hay quên! }, stats: () => ({ size: store.size, hits, misses, hitRate: hits + misses === 0 ? 0 : Math.round(hits / (hits + misses) * 100) }) }; } // ── Sử dụng ────────────────────────────────────── const cache = createCache({ ttl: 5000, maxSize: 3 }); cache.set('A', 'Alpha'); // Map: [A] A=MRU cache.set('B', 'Beta'); // Map: [A, B] B=MRU cache.set('C', 'Gamma'); // Map: [A, B, C] C=MRU, A=LRU cache.get('A'); // HIT + promote → Map: [B, C, A] A=MRU cache.set('D', 'Delta'); // Map đầy! Evict LRU=B → Map: [C, A, D] cache.get('B'); // null — B đã bị evict! cache.stats(); // { size: 3, hits: 1, misses: 1, hitRate: 50 }
Tại sao Map là cấu trúc hoàn hảo cho LRU? JavaScript Map bảo toàn thứ tự insertion theo chuẩn ES2015+. Sau delete + set, key đó nằm ở cuối iterator. Vì vậy: đầu Map = LRU (ít dùng nhất), cuối Map = MRU (vừa dùng). Không cần Doubly Linked List như LRU trong Java/C++!

Câu hỏi Phỏng vấn

Câu hỏiĐiểm cốt lõi cần trả lời
"Closure là gì?" Function + lexical environment. Giải thích bằng ví dụ counter. Đề cập Closure Environment tồn tại dù outer function đã kết thúc.
"Tại sao var trong for loop gây bug?" var không có block scope → tất cả function closure over cùng 1 biến i. 3 cách fix: let, IIFE, factory function.
"Event Loop hoạt động thế nào?" Single thread. Thứ tự: Sync → Microtask Queue → Macrotask Queue. Promise .then là microtask, setTimeout là macrotask.
"Promise.all vs allSettled?" all: fail-fast (1 fail → all fail). allSettled: luôn resolve, báo kết quả từng cái. Dùng all khi cần tất cả; allSettled khi muốn biết cái nào fail.
"async/await có phải Promise không?" Syntactic sugar. async fn luôn trả Promise. await nhường control về Event Loop (không block thread). return await trong try/catch khác return.
"Implement debounce từ đầu?" Closure giữ timer. Mỗi lần gọi: clearTimeout(timer), đặt timer mới. Phải giải thích tại sao closure cần thiết ở đây.
"Race condition trong async là gì?" Request cũ về sau request mới, ghi đè kết quả sai. Fix: AbortController hoặc Request ID pattern dùng closure.
"LRU Cache hoạt động thế nào?" Map giữ thứ tự insertion. get() phải promote (delete + re-insert). Evict phần tử đầu Map (LRU). TTL check bằng createdAt. clear() phải reset stats.

Cheat Sheet

CLOSURE
  • = function + lexical environment
  • Biến sống dù outer fn đã kết thúc
  • var loop → dùng let hoặc factory
  • Mỗi lần gọi factory → closure riêng
  • Private state không cần class
  • Stale closure trong async → dùng ref
PROMISE
  • 3 state: pending → fulfilled / rejected
  • Chỉ chuyển state 1 lần, không đảo ngược
  • fetch() không throw khi HTTP 4xx/5xx!
  • forEach + async ≠ await tuần tự
  • return await (có await) trong try/catch
  • Micro trước Macro (Promise trước setTimeout)
PATTERNS
  • Debounce: gọi sau khi yên X ms
  • Throttle: tối đa 1 lần / X ms
  • Memoize: cache theo args (JSON key)
  • LRU: Map, promote = delete+re-insert
  • Retry: exponential backoff 2^n * base
  • Race fix: AbortController / Request ID
COMMON PITFALLS
  • var trong loop → FIFO bug
  • await trong forEach → không đúng ý
  • Sequential await khi có thể parallel
  • LRU không promote → thực ra là FIFO
  • clear() không reset hits/misses
  • Race condition search → stale result
