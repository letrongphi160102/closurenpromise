Module 1 · Closure

Closure

1.1 Execution Context & Call Stack

Cơ bản

Mỗi khi JavaScript chạy một function, engine tạo ra một Execution Context (EC) — một "hộp cát" riêng chứa: biến cục bộ, tham chiếu đến scope cha, và con trỏ this. Tất cả EC được quản lý bởi Call Stack theo nguyên tắc LIFO (Last In, First Out).

JavaScriptconst x = 10; // Global EC: x = 10 function outer() { const y = 20; // outer() EC: y = 20 function inner() { const z = 30; // inner() EC: z = 30 console.log(x + y + z); // 60 } inner(); } outer();
Thời điểmCall StackBiến tồn tại
Khởi động[Global EC]x = 10
outer() được gọi[Global EC, outer EC]x = 10, y = 20
inner() được gọi[Global EC, outer EC, inner EC]x, y, z đều tồn tại
inner() kết thúc[Global EC, outer EC]z bị hủy
outer() kết thúc[Global EC]y bị hủy
Tại sao cần hiểu EC? Closure hoạt động dựa trên việc EC "bị giữ lại" dù đã pop khỏi Call Stack. Hiểu EC là nền tảng để hiểu closure không phải là "magic" — nó có cơ chế cụ thể.

1.2 Scope: local / global / block

Cơ bản

Scope là phạm vi mà một biến có thể được truy cập. JavaScript có 3 loại scope quan trọng, và sự khác biệt giữa var, let, const nằm chính ở đây.

Global Scope

var globalVar = 'everywhere'; let globalLet = 'everywhere'; const globalConst = 'everywhere'; // Tất cả đều truy cập được từ mọi nơi // var còn gắn vào window object: window.globalVar; // ✅ 'everywhere' window.globalLet; // undefined window.globalConst; // undefined

Function Scope (Local)

function myFunc() { var a = 1; // chỉ trong myFunc let b = 2; // chỉ trong myFunc const c = 3; // chỉ trong myFunc } console.log(a); // ❌ ReferenceError console.log(b); // ❌ ReferenceError console.log(c); // ❌ ReferenceError

Block Scope — Sự khác biệt then chốt giữa var và let/const

❌ var — KHÔNG có block scope
if (true) { var x = 10; // "thoát" ra ngoài block! } console.log(x); // ✅ 10 (var bị hoisted) for (var i = 0; i < 3; i++) {} console.log(i); // ✅ 3 (tồn tại sau loop!) // var chỉ bị giới hạn bởi FUNCTION scope // không bị giới hạn bởi {} block
✅ let / const — có block scope
if (true) { let y = 20; // chỉ sống trong { } const z = 30; // chỉ sống trong { } } console.log(y); // ❌ ReferenceError console.log(z); // ❌ ReferenceError for (let i = 0; i < 3; i++) {} console.log(i); // ❌ ReferenceError // let/const bị giới hạn bởi bất kỳ {} nào

Hoisting — var vs let/const

// var được hoisted và khởi tạo = undefined: console.log(a); // undefined (không phải lỗi!) var a = 5; console.log(a); // 5 // let/const được hoisted nhưng KHÔNG khởi tạo → Temporal Dead Zone: console.log(b); // ❌ ReferenceError: Cannot access 'b' before initialization let b = 5;

1.3 Lexical Scope & Scope Chain

Cơ bản

Lexical scope (còn gọi là static scope) có nghĩa là: scope của một biến được xác định tại lúc viết code, không phải lúc chạy. JavaScript nhìn vào nơi function được định nghĩa để tìm biến — không phải nơi nó được gọi.

❌ Tưởng dynamic scope
const name = 'Global'; function greet() { // greet được ĐỊNH NGHĨA ở Global // nên nó đọc name từ Global scope console.log(name); } function wrapper() { const name = 'Local'; greet(); // gọi từ đây } // Bạn nghĩ sẽ in 'Local'? wrapper(); // → 'Global' (không phải 'Local'!)
✅ Đây là lexical scope
// JS xác định scope lúc VIẾT CODE: // greet() được viết ở Global scope // → nó thấy name = 'Global' // dù được GỌI từ trong wrapper() function outer() { const msg = 'from outer'; function inner() { // inner được định nghĩa trong outer // → thấy msg của outer console.log(msg); // 'from outer' } return inner; } const fn = outer(); fn(); // vẫn → 'from outer'

Scope Chain — Chuỗi tìm kiếm biến

Khi tìm một biến, JavaScript tìm từ scope hiện tại → lên scope cha → lên tiếp cho đến Global. Nếu không tìm thấy ở Global → ReferenceError.

const a = 1; // Global scope function level1() { const b = 2; // level1 scope function level2() { const c = 3; // level2 scope function level3() { // Tìm d: level3 → level2 → level1 → global → ReferenceError // Tìm c: level3 → level2 ✅ (tìm thấy) // Tìm b: level3 → level2 → level1 ✅ // Tìm a: level3 → level2 → level1 → global ✅ console.log(a + b + c); // 6 } level3(); } level2(); }
Scope chain chỉ đi lên, không đi xuống! Scope cha không thể truy cập biến của scope con. level1 không thể đọc c hay d. Đây là tính đóng gói tự nhiên của JavaScript.

1.4 Closure — Bản chất hoạt động

Trung cấp

Định nghĩa chính xác: Closure là khả năng một function ghi nhớ và tiếp cận lexical scope của nó ngay cả khi function đó đang thực thi bên ngoài scope ban đầu.

Nói cụ thể hơn: khi outer() kết thúc, EC của nó bị pop khỏi Call Stack. Theo lẽ thường, các biến bên trong sẽ bị garbage collect. Nhưng nếu hàm bên trong vẫn còn tham chiếu đến chúng — JavaScript engine giữ những biến đó lại trong Closure Record.

function makeCounter() { let count = 0; // sẽ được giữ trong Closure Record return function increment() { count++; // vẫn truy cập được count! return count; }; } const counter = makeCounter(); // makeCounter() đã kết thúc và EC đã bị pop // nhưng biến count vẫn còn sống trong Closure Record counter(); // → 1 counter(); // → 2 counter(); // → 3 // Không thể truy cập count trực tiếp từ ngoài: counter.count; // undefined

Mỗi lần gọi = một Closure Environment riêng biệt

const c1 = makeCounter(); // Closure Record 1: { count: 0 } const c2 = makeCounter(); // Closure Record 2: { count: 0 } — hoàn toàn riêng biệt! c1(); // → 1 c1(); // → 2 c2(); // → 1 (count của c2 không bị ảnh hưởng bởi c1!) c1(); // → 3 c2(); // → 2
Closure = function + snapshot của lexical environment tại thời điểm tạo ra nó. Không phải snapshot của giá trị — mà là snapshot của tham chiếu đến biến. Vì vậy nếu biến thay đổi sau khi closure được tạo, closure thấy giá trị mới nhất.
// Closure giữ THAM CHIẾU đến biến, không phải GIÁ TRỊ: function demo() { let x = 1; const getX = () => x; x = 99; // thay đổi sau khi tạo closure return getX; } demo()(); // → 99 (không phải 1!)

1.5 Closure trong vòng lặp — Lỗi kinh điển

Trung cấp

Đây là bẫy phỏng vấn số 1. Vì closure giữ tham chiếu đến biến (không phải giá trị), khi dùng var trong loop, tất cả closure cùng trỏ vào một biến duy nhất.

❌ Bug — dùng var
var funcs = []; for (var i = 0; i < 3; i++) { funcs[i] = function() { return i; // closure over biến i duy nhất }; } // Sau loop: i = 3 funcs[0](); // → 3 (không phải 0!) funcs[1](); // → 3 funcs[2](); // → 3
✅ Fix 1 — dùng let
const funcs = []; for (let i = 0; i < 3; i++) { // let tạo binding MỚI mỗi iteration funcs[i] = function() { return i; // mỗi closure có i riêng }; } funcs[0](); // → 0 ✓ funcs[1](); // → 1 ✓ funcs[2](); // → 2 ✓
✅ Fix 2 — IIFE
var funcs = []; for (var i = 0; i < 3; i++) { funcs[i] = (function(j) { // j là parameter — value copy của i // mỗi IIFE có j riêng biệt return function() { return j; }; })(i); // gọi ngay, truyền i vào }
✅ Fix 3 — Factory function (sạch nhất)
function createFn(n) { return function() { return n; }; } var funcs = []; for (var i = 0; i < 3; i++) { funcs[i] = createFn(i); } // n là parameter, không bị chia sẻ
Tại sao var bị lỗi? var không có block scope. Toàn bộ vòng lặp chỉ có một biến i dùng chung. Các closure không lưu giá trị i — chúng lưu tham chiếu đến biến i. Khi loop kết thúc, i = 3, tất cả closure đọc cùng một giá trị đó.

1.6 Factory Function & Module Pattern

Nâng cao

Factory function là hàm trả về object với private state qua closure. Đây là cách implement encapsulation trong JavaScript mà không cần class.

// Factory Function — private state + public interface function createUser(name, role) { // PRIVATE — chỉ các method bên dưới thấy được let loginCount = 0; const createdAt = new Date(); // PUBLIC interface return { name, role, login() { loginCount++; return `${name} đăng nhập lần #${loginCount}`; }, getStats: () => ({ loginCount, createdAt }) }; } const alice = createUser('Alice', 'admin'); const bob = createUser('Bob', 'user'); alice.login(); // → "Alice đăng nhập lần #1" alice.login(); // → "Alice đăng nhập lần #2" bob.login(); // → "Bob đăng nhập lần #1" (độc lập!) alice.loginCount; // undefined — không thể truy cập trực tiếp!

Factory vs Class — Vấn đề this binding

❌ Class — this dễ mất
class Counter { constructor() { this.n = 0; } inc() { this.n++; return this.n; } } const c = new Counter(); const { inc } = c; inc(); // 💥 TypeError: Cannot read // properties of undefined // Phải bind: c.inc.bind(c) // Hoặc arrow function trong class
✅ Factory — không cần this
function createCounter() { let n = 0; return { inc: () => ++n, // arrow fn, dùng closure dec: () => --n, get: () => n }; } const c = createCounter(); const { inc } = c; inc(); inc(); inc(); c.get(); // ✅ 3 — hoạt động đúng

Revealing Module Pattern

// IIFE + closure = module với public/private API rõ ràng const CartModule = (() => { // PRIVATE let items = []; let discount = 0; function calcTotal() { const subtotal = items.reduce((sum, item) => sum + item.price, 0); return subtotal * (1 - discount); } // PUBLIC — chỉ expose những gì cần thiết return { addItem(item) { items.push(item); }, removeItem(id) { items = items.filter(i => i.id !== id); }, setDiscount(d) { discount = Math.min(d, 1); }, getTotal() { return calcTotal(); }, getCount: () => items.length }; })(); // IIFE — chạy ngay, trả về object CartModule.addItem({ id: 1, price: 100 }); CartModule.setDiscount(0.1); CartModule.getTotal(); // → 90 CartModule.items; // undefined — protected!

Composition thay vì Inheritance

// Tách thành các "capability" nhỏ, ghép lại tùy ý const canSwim = (s) => ({ swim: () => `${s.name} đang bơi` }); const canFly = (s) => ({ fly: () => `${s.name} đang bay` }); const canDive = (s) => ({ dive: () => `${s.name} đang lặn` }); const createDuck = (name) => Object.assign({name}, canSwim({name}), canFly({name}), canDive({name})); const createPenguin = (name) => Object.assign({name}, canSwim({name}), canDive({name})); // penguin không có canFly — không cần sửa class hierarchy!

1.7 Memoization & Private state

Nâng cao

Memoization là kỹ thuật dùng closure để cache kết quả tính toán tốn kém. Lần đầu gọi → tính và lưu. Lần sau gọi cùng args → trả từ cache ngay.

function memoize(fn) { const cache = new Map(); // closure — sống suốt vòng đời của memoized fn return function(...args) { const key = JSON.stringify(args); if (cache.has(key)) { console.log('Cache hit!'); return cache.get(key); } const result = fn(...args); cache.set(key, result); return result; }; } // Fibonacci không tối ưu: O(2^n) const fib = memoize(function _fib(n) { return n <= 1 ? n : fib(n - 1) + fib(n - 2); }); console.log(fib(40)); // tính lần đầu (chậm) console.log(fib(40)); // cache hit — tức thì! console.log(fib(39)); // cache hit — đã tính khi tính fib(40)

Partial Application

// Closure giữ "preset args" — tạo hàm chuyên dụng từ hàm tổng quát function partial(fn, ...presetArgs) { return function(...laterArgs) { // closure giữ presetArgs return fn(...presetArgs, ...laterArgs); }; } const multiply = (a, b) => a * b; const double = partial(multiply, 2); const triple = partial(multiply, 3); double(7); // → 14 triple(7); // → 21 // Real-world: API với base URL cố định const apiFetch = (method, url, data) => fetch(url, { method, body: JSON.stringify(data) }); const GET = partial(apiFetch, 'GET'); const POST = partial(apiFetch, 'POST'); GET('/users'); // → apiFetch('GET', '/users', undefined) POST('/users', userData); // → apiFetch('POST', '/users', userData)

Promise & Async

2.1 Synchronous vs Asynchronous

Cơ bản

Đây là nền tảng để hiểu tại sao Promise và async/await ra đời. JavaScript là single-threaded — chỉ làm một việc tại một thời điểm. Nhưng web cần làm nhiều việc cùng lúc (gọi API, đọc file, đếm ngược...).

Synchronous — tuần tự, chặn nhau
// Sync: mỗi dòng chờ dòng trước hoàn thành console.log('Bắt đầu'); const result = heavyCalculation(); // ⛔ BLOCK 5 giây // Trong 5 giây này: UI đóng băng, // user không click được, scroll không được console.log('Xong:', result); // sau 5s
Asynchronous — không chặn
// Async: giao việc đi, tiếp tục làm việc khác console.log('Bắt đầu'); fetch('/api/data') // giao cho browser, không chờ .then(res => res.json()) .then(data => console.log('Nhận data:', data)); console.log('Dòng này chạy ngay!'); // không chờ fetch // UI vẫn phản hồi trong khi chờ API

Cơ chế: Web APIs + Callback Queue

JavaScript không tự xử lý async — nó giao việc cho môi trường (browser hoặc Node.js). Browser có các Web APIs chạy trên thread riêng: setTimeout, fetch, DOM events, v.v. Khi xong, chúng đẩy callback vào queue để JS xử lý sau.

// Sơ đồ luồng async: // // JS Code → Call Stack → Web APIs (browser thread riêng) // ↓ (khi xong) // ←←←←←←←←←← Event Loop ← Callback Queue // // Event Loop: "Nếu Call Stack trống → lấy callback từ Queue để chạy" setTimeout(() => console.log('Sau 0ms'), 0); // Dù delay = 0, vẫn không chạy ngay // Phải chờ Call Stack trống trước! console.log('Tôi chạy trước'); // chạy trước dù setTimeout(0)

2.2 Event Loop & Callback Queue

Cơ bản

Event Loop là vòng lặp liên tục kiểm tra: "Call Stack trống chưa? Nếu rồi thì lấy task tiếp theo từ queue". Có 2 loại queue với độ ưu tiên khác nhau.

// Câu đố kinh điển — đoán output: console.log('1 — sync'); setTimeout(() => console.log('2 — macrotask'), 0); Promise.resolve().then(() => console.log('3 — microtask')); queueMicrotask(() => console.log('4 — microtask 2')); console.log('5 — sync'); // Output: 1 → 5 → 3 → 4 → 2 // Thứ tự: Sync → Microtask Queue (hết) → Macrotask Queue
Microtask QueueMacrotask Queue
Bao gồmPromise .then/.catch, queueMicrotask, MutationObserversetTimeout, setInterval, I/O, UI events
Độ ưu tiênCao hơn — chạy trướcThấp hơn — chạy sau
DrainXử lý hết toàn bộ queue trước khi sang macrotaskMỗi vòng Event Loop chỉ lấy 1 task
// Nguy hiểm: Microtask vô hạn → block macrotask mãi mãi! function infiniteMicro() { Promise.resolve().then(infiniteMicro); // đừng làm thế này! } // UI sẽ đóng băng vì setTimeout không bao giờ chạy được
Quy tắc vàng: Toàn bộ Microtask Queue được drain hết trước khi Event Loop lấy task tiếp từ Macrotask Queue. Vì vậy Promise.then luôn chạy trước setTimeout dù cả hai đều "pending".

2.3 Callback Hell — Vấn đề cần giải quyết

Cơ bản

Khi cần thực hiện nhiều tác vụ async tuần tự (kết quả sau phụ thuộc vào kết quả trước), callback lồng vào nhau tạo ra "Pyramid of Doom".

❌ Callback Hell
getUser(id, function(err, user) { if (err) { handleError(err); return; } getPosts(user.id, function(err, posts) { if (err) { handleError(err); return; } getComments(posts[0].id, function(err, comments) { if (err) { handleError(err); return; } saveLog(comments, function(err) { if (err) { handleError(err); return; } console.log('Xong!'); // level 4! }); }); }); });
Vấn đề của callback hell:

❌ Khó đọc — indent tăng mãi
❌ Error handling lặp ở mỗi bước
❌ Không dùng được try/catch
❌ Khó chạy song song
❌ Khó test và debug
❌ Dễ quên return sau handleError

Trung cấp

Promise là object đại diện cho một giá trị chưa có ngay. Nó có đúng 3 trạng thái và chỉ chuyển trạng thái một lần duy nhất, không thể đảo ngược.

// Tạo Promise thủ công với new Promise(executor) const p = new Promise((resolve, reject) => { // executor chạy ngay lập tức (synchronous) // resolve() hoặc reject() chỉ gọi 1 lần — lần sau bị bỏ qua setTimeout(() => { const ok = Math.random() > 0.5; if (ok) resolve('Thành công'); // → Fulfilled else reject(new Error('Thất bại')); // → Rejected }, 1000); }); // Shorthand cho Promise đã settled ngay: Promise.resolve(42); // Fulfilled với value = 42 Promise.reject(new Error()); // Rejected với error

2.5 .then() / .catch() / .finally()

Trung cấp

Đây là bộ 3 method để xử lý kết quả từ Promise. Chúng trả về Promise mới — cho phép chain liên tiếp.

// .then(onFulfilled, onRejected) — handler cho cả 2 trường hợp // .catch(onRejected) — shorthand: .then(undefined, onRejected) // .finally(onFinally) — luôn chạy dù fulfilled hay rejected fetch('/api/users/1') .then(res => { // ⚠️ fetch() CHỈ reject khi network lỗi! // HTTP 404, 500 vẫn là Fulfilled — phải tự check: if (!res.ok) throw new Error(`HTTP ${res.status}`); return res.json(); // trả Promise → chain tiếp }) .then(user => { console.log('User:', user); return getPostsByUser(user.id); // trả Promise → chain tiếp }) .then(posts => { console.log('Posts:', posts); }) .catch(err => { // Bắt LỖI từ BẤT KỲ bước nào trong chain ở trên console.error('Lỗi:', err.message); }) .finally(() => { // Luôn chạy — dùng để dọn dẹp (ẩn loading spinner, v.v.) hideLoadingSpinner(); });

Cơ chế chaining — quan trọng phải hiểu rõ

// Mỗi .then() trả về một Promise MỚI // Giá trị return trong .then() trở thành value của Promise mới đó: Promise.resolve(1) .then(x => x + 1) // → 2 .then(x => x * 10) // → 20 .then(x => { if (x > 15) throw new Error('Quá lớn'); // throw → rejected return x; }) .catch(err => { console.error(err.message); // 'Quá lớn' return 0; // catch CÓ THỂ recover! trả giá trị → fulfilled lại }) .then(x => console.log('Final:', x)); // 'Final: 0'
❌ Anti-pattern: Promise Hell
// Lồng .then() — tạo ra pyramid tương tự callback! getUser(id).then(user => { getPosts(user.id).then(posts => { getComments(posts[0].id).then(comments => { console.log(comments); // không thể catch tập trung! }); }); });
✅ Đúng: Flat chain
// Trả Promise trong .then() → chain phẳng getUser(id) .then(user => getPosts(user.id)) .then(posts => getComments(posts[0].id)) .then(comments => { console.log(comments); }) .catch(err => handleError(err)); // ✅ Một .catch() cho tất cả!

2.7 async / await — Sugar syntax

Nâng cao

async/await là syntactic sugar trên Promise — không phải tính năng mới hoàn toàn. Một async function luôn trả về Promise. await không block JS thread — nó nhường control về Event Loop và tiếp tục khi Promise settle.

// Dưới đây là 2 cách viết TƯƠNG ĐƯƠNG: // Cách 1: Promise chain function loadUserData(id) { return getUser(id) .then(user => getPosts(user.id)) .then(posts => posts[0]); } // Cách 2: async/await (đọc như code đồng bộ) async function loadUserData(id) { const user = await getUser(id); const posts = await getPosts(user.id); return posts[0]; }

❌ 3 lỗi phổ biến

// ❌ Race condition: request cũ về sau request mới → ghi đè kết quả sai async function searchBuggy(query) { const results = await fetchSearch(query); showResults(results); // 💥 "cat" (chậm) có thể ghi đè "cats" (nhanh) } // ✅ Fix 1: AbortController — hủy request cũ trước khi gửi request mới let controller = null; async function searchAbort(query) { controller?.abort(); // hủy request đang chạy controller = new AbortController(); try { const res = await fetch(`/search?q=${query}`, { signal: controller.signal }); showResults(await res.json()); } catch (e) { if (e.name !== 'AbortError') throw e; // bỏ qua AbortError — là bình thường } } // ✅ Fix 2: Request ID pattern (dùng closure) let latestId = 0; async function searchById(query) { const thisId = ++latestId; // closure capture ID này const results = await fetchSearch(query); if (thisId !== latestId) return; // request cũ hơn → bỏ qua showResults(results); // ✅ chỉ hiện kết quả mới nhất }

3.2 Retry Pattern với Closure

Nâng cao
function withRetry(fn, maxRetries = 3, baseDelay = 100) { // closure giữ fn, maxRetries, baseDelay return async function(...args) { let attempt = 0; while (attempt < maxRetries) { try { return await fn(...args); // thành công → return ngay } catch (err) { attempt++; if (attempt === maxRetries) throw err; // hết lần thử → throw // Exponential backoff: 100ms → 200ms → 400ms const delay = Math.pow(2, attempt) * baseDelay; console.warn(`Thử lại lần ${attempt}/${maxRetries} sau ${delay}ms`); await new Promise(r => setTimeout(r, delay)); } } }; } // Sử dụng: const fetchWithRetry = withRetry(fetch, 3); const res = await fetchWithRetry('/api/flaky-endpoint'); // Kết hợp với timeout: const robustFetch = withRetry( (url) => Promise.race([ fetch(url), new Promise((_, r) => setTimeout(() => r(new Error('Timeout')), 5000)) ]), 3 );

3.3 Debounce & Throttle từ đầu

Nâng cao
Phân biệt: Debounce = "chờ yên X ms rồi mới chạy" — reset timer mỗi lần gọi. Throttle = "tối đa 1 lần mỗi X ms" — bỏ qua nếu chưa đủ thời gian. Cả hai đều dùng closure để giữ state giữa các lần gọi.

Debounce

function debounce(fn, delay) { let timer; // closure! return function(...args) { clearTimeout(timer); // reset timer mỗi lần gọi timer = setTimeout(() => { fn.apply(this, args); }, delay); }; } // Search input — gọi API sau khi dừng gõ 500ms const onSearch = debounce( (query) => callSearchAPI(query), 500 ); input.addEventListener('input', (e) => onSearch(e.target.value)); // Gõ "h" → "he" → "hel" → "hell" → "hello" // Chỉ gọi API 1 lần với "hello" (sau 500ms dừng)

Throttle

function throttle(fn, limit) { let lastRun = 0; // closure! return function(...args) { const now = Date.now(); if (now - lastRun >= limit) { lastRun = now; fn.apply(this, args); } // chưa đủ thời gian → bỏ qua call này }; } // Scroll handler — update UI tối đa 3.3 lần/giây const onScroll = throttle( () => updateProgressBar(), 300 ); window.addEventListener('scroll', onScroll); // Scroll liên tục → chạy mỗi 300ms một lần

3.4 Xây dựng LRU Cache với TTL

Nâng cao
LRU = Least Recently Used. Khi cache đầy và cần thêm item mới, xóa item ít được truy cập gần đây nhất. Lỗi phổ biến: dùng Map nhưng không promote sau get() → thực chất là FIFO, không phải LRU.

Tại sao Map là cấu trúc hoàn hảo cho LRU?

JavaScript Map bảo toàn thứ tự insertion theo chuẩn ES2015+. Khi ta delete rồi set lại một key, nó chuyển về cuối iterator. Vì vậy: đầu Map = LRU (ít dùng nhất), cuối Map = MRU (vừa dùng).

function createCache({ ttl = 10000, maxSize = 100 } = {}) { // CLOSURE — toàn bộ state là private const store = new Map(); let hits = 0, misses = 0; const isExpired = (entry) => Date.now() - entry.createdAt > ttl; // ✅ Cốt lõi LRU: delete + re-insert = move về cuối Map (MRU) function promote(key) { const entry = store.get(key); store.delete(key); store.set(key, entry); // về cuối = MRU position } return { get(key) { const e = store.get(key); if (!e || isExpired(e)) { store.delete(key); misses++; return null; } promote(key); // ✅ Phải promote sau mỗi get! hits++; return e.value; }, set(key, value) { if (store.has(key)) { store.delete(key); // ✅ Xóa vị trí cũ để re-insert về MRU } else if (store.size >= maxSize) { // ✅ Evict đúng LRU = phần tử ĐẦU TIÊN của Map const lruKey = store.keys().next().value; store.delete(lruKey); } store.set(key, { value, createdAt: Date.now() }); }, invalidate(key) { store.delete(key); }, clear() { store.clear(); hits = 0; misses = 0; // ✅ Reset stats! }, stats: () => ({ size: store.size, hits, misses, hitRate: hits + misses === 0 ? 0 : Math.round(hits / (hits + misses) * 100) }) }; } // ── Demo thứ tự LRU ────────────────────────────────────────── const cache = createCache({ ttl: 5000, maxSize: 3 }); cache.set('A', 1); // Map: [A] A=LRU=MRU cache.set('B', 2); // Map: [A, B] A=LRU, B=MRU cache.set('C', 3); // Map: [A, B, C] A=LRU, C=MRU — đầy rồi! cache.get('A'); // HIT → promote A Map: [B, C, A] B=LRU, A=MRU cache.set('D', 4); // đầy! evict LRU=B Map: [C, A, D] C=LRU, D=MRU cache.get('B'); // → null (B đã bị evict) cache.stats(); // { size:3, hits:1, misses:1, hitRate:50 }

Câu hỏi Phỏng vấn Thường gặp

Câu hỏiĐiểm cốt lõi cần trả lời
"Closure là gì?"Function + lexical environment. EC của outer fn bị pop nhưng biến được giữ trong Closure Record nếu còn tham chiếu. Ví dụ counter.
"var vs let vs const?"var: function scope, hoisted=undefined, gắn window. let: block scope, TDZ. const: block scope, TDZ, không reassign. Khi nào dùng cái nào.
"Tại sao var trong for loop bị lỗi?"var không có block scope → 1 biến i duy nhất. Closure giữ reference, không phải value. Khi chạy, i=3. Fix: let / IIFE / factory.
"Event Loop là gì?"Single-thread. Sync → Microtask Queue (drain hết) → Macrotask Queue (1 task). Promise.then = micro, setTimeout = macro.
"Promise.all vs allSettled?"all: fail-fast, 1 reject → tất cả reject. allSettled: luôn resolve, báo cáo từng kết quả. Dùng all khi cần tất cả; allSettled khi muốn xử lý từng cái.
"return vs return await trong try/catch?"return promise: lỗi xảy ra SAU khi thoát try → catch không bắt được. return await: lỗi xảy ra BÊN TRONG try → catch bắt được.
"Implement debounce từ đầu?"Closure giữ timer. clearTimeout mỗi lần gọi, setTimeout mới. Giải thích tại sao closure cần thiết (timer tồn tại giữa các lần gọi).
"LRU Cache hoạt động thế nào?"Map giữ thứ tự insertion. get() phải promote (delete + re-insert = về cuối = MRU). set() check has() trước. Evict = keys().next() = đầu Map = LRU. clear() reset stats.

Cheat Sheet

CLOSURE — Kim cương nhớ mãi
  • = function + lexical environment
  • Biến sống dù outer EC đã bị pop
  • Giữ reference đến biến, không phải value
  • var loop bug → dùng let hoặc factory
  • Mỗi lần gọi factory = closure environment riêng
  • Stale closure trong async → dùng object ref
SCOPE — Nền tảng
  • var: function scope, hoisted=undefined
  • let/const: block scope, TDZ hoisting
  • Lexical: scope xác định khi VIẾT code
  • Scope chain: inner → outer → global
  • Chỉ đi lên, không đi xuống
  • Default: dùng const, let khi cần reassign
PROMISE — Async chuyên nghiệp
  • 3 state: pending → fulfilled / rejected (1 lần)
  • fetch() không throw khi HTTP 4xx/5xx!
  • .catch() trong chain có thể recover
  • Micro (Promise) trước Macro (setTimeout)
  • forEach + async ≠ sequential
  • return await (có await) trong try/catch!
PATTERNS — Thực chiến
  • Debounce: chờ yên X ms → 1 lần cuối
  • Throttle: tối đa 1 lần / X ms
  • Memoize: cache theo JSON.stringify(args)
  • LRU: promote = delete + re-insert (MRU)
  • Retry: exponential backoff 2^n × base
  • Race fix: AbortController / Request ID
